# 高层设计

本解决方案设计文档将帮助我们探索和了解 OpenMetadata 服务的内部结构、构建方式及其交互。

我们将首先描述应用程序软件设计的全貌。我们将一点一点地深入具体组件，描述它们的行为，并举例说明如何使用它们。

## 系统背景

第一部分的目的是熟悉高级概念和相关技术。学习目标如下

- 描述组成 OpenMetadata 的元素及其关系。
- 终端用户和外部应用程序如何与系统通信。

这里我们有解决方案的主要参与者：

![system-context](../../../images/v1.5/main-concepts/high-level-design/system-context.png/)

- 应用程序接口（API）： 这是 OpenMetadata 的主要支柱。在这里，我们定义了如何与元数据实体交互。
  - 它为解决方案的所有其他组件提供动力。
- 用户界面(UI)： 以发现为中心的工具，可帮助用户跟踪组织中的所有数据资产。其目标是促进和推动协作。
- 摄取框架(Ingestion Framework)： 该系统基于应用程序接口规范，是所有连接器（即定义 OpenMetadata 与包含我们要集成的元数据的外部系统之间的交互的组件）的基础。
- 实体存储(Entity Store)： MySQL 存储，包含所有实体及其关系状态的实时信息。
- 搜索引擎(Search Engine)： 由 ElasticSearch 提供支持，是用户界面的索引系统，可帮助用户发现元数据。

## JSON 模式

如果我们暂时抽象出存储层，就会意识到 OpenMetadata 的实现是三个模块的集成：

- 核心应用程序接口（API），统一和集中与内部和外部系统的通信。
- 以团队为中心的元数据服务层的用户界面(UI)。
- 摄取框架(Ingestion Framework)，作为 OpenMetadata 与外部资源之间的接口。

这些组件的唯一共同点是词汇-->它们都在塑造、描述和移动元数据实体。

OpenMetadata 基于元数据的标准定义。因此，我们需要确保在实施这一标准时，端到端工作流中共享这一定义。为此，我们将主要词库定义为 JSON 模式，这是一种可读且与语言无关的解决方案。

然后，在打包主要组件时，我们为所有实体生成特定的编程类。
我们从同一源头实现了三种视图：

- 用于应用程序接口的 Java 类
- 用于摄取框架的 Python 类
- 用户界面的 TypeScript 类型

它们中的每一个都以单一的真实源为模型。由于采用了这种方法，我们可以确信，无论我们在整个过程中放大到哪个点，我们总能找到一个统一的定义明确的实体。

## API 容器图

现在，我们将放大 API 容器。作为解决方案的中央软件系统，它的目标是管理调用（包括来自内部和外部的调用，如摄取框架或任何自定义集成），并更新
元数据实体的状态。

当数据存储在 MySQL 容器中时，API 将负责获取数据并完成必要的信息、验证实体数据和所有关系。

服务层（API）与存储层分离后，用户和集成可以用简单的语言（REST）获取所需信息，而无需深入学习特定的数据模型和设计选择。

![api-container-diagram](../../../images/v1.5/main-concepts/high-level-design/api-container-diagram.png)

## 实体资源

当我们与大多数实体交互时，我们遵循相同的端点结构。例如

- `GET <url>/api/v1/<collectionName>/<id>` 按 ID 检索实体实例。
- `GET <url>/api/v1/<collectionName>/name/<FQN>` 按完全合格域名查询。

同样，我们还支持其他 CRUD 操作，每种操作都需要一个特定的传入数据结构，并返回实体的类。由于 OpenMetadata 的基础是实体定义，因此我们与任何消费者都有这种数据契约，后端将验证接收到的数据以及输出结果。

端点定义和数据类型设置在实体资源中进行。每个元数据实体都包含一个资源类，该类为给定的实体构建 API 定义。

这种逻辑随后会出现在 [API](/swagger.html) 文档中。

## 实体存储库

实体资源库的目标是对后端数据库(backend database)执行读写操作，以创建、检索、更新和删除实体。

实体资源负责处理外部通信，而存储库则负责管理整个流程与存储层的交互，确保传入和传出的实体是有效的，并持有正确和完整的信息。

这意味着，我们要在这里定义 DAO（数据访问对象）以及所有验证和数据存储逻辑。

由于在所有实体中都有重复的过程（例如，在集合中列出实体或从实体中获取特定版本），因此实体存储库扩展了一个接口(Interface)，该接口实现了一些基本功能，并抽象了实体的特定逻辑。

然后，每个实体需要实现其服务器端流程(server-side processes)，如根据实体层次结构构建 FQN，实体如何存储和检索与其他实体的关系(Relationship)信息，或实体如何对变更事件(Change Events)做出反应。

## 实体存储层

在 “API容器图(API Container Diagram)” 中，我们展示了实体存储库如何根据处理的信息类型与三种不同的存储容器（表）进行交互。

要充分理解这一决定，我们首先要谈谈实体实例所包含的信息。

实体有两类字段：属性（JSON Schema 属性）和关系（JSON Schema href）：

- 属性(Attributes)是实体的核心属性：名称和 id、表格的列或 ML 模型的算法。这些都是实体的固有信息，它们的存在和值有助于我们区分实体实例（表 A 与表 B）和实体定义（仪表盘与主题）。
- 关系(Relationships)是两个实体之间的关联。例如，表属于数据库，用户拥有仪表盘等。关系是一种特殊类型的属性，使用实体引用来捕获。

## 实体和关系存储

实体以 JSON 文档的形式存储在数据库中。每个实体都有一个关联表（`<entityName>_entity`），其中包含定义实体属性和其他元数据字段（如 id、`updatedAt` 或 `updatedBy`）的 JSON。

该 JSON 不存储任何关系。例如，拥有仪表盘的用户是一条信息，它在单独的表 entity_relationship 中以图节点的形式具体化，其中的边代表关系的类型（`contains`, `uses`, `follows`......）。

这种分离有助于我们分清关注点。我们可以独立处理相关实体，并在运行时验证哪些信息需要更新和/或检索。例如，如果我们删除了用户拥有的仪表板，那么我们将清理实体关系(entity_relationship)中的这一行，但这不会改变用户的信息。

另一个更棘手的例子是删除包含表的数据库。在这种情况下，程序会检查数据库实体是否为空，因此我们无法继续删除。

## 更改事件存储

您可能已经注意到，在所有实体定义中，我们都有一个 changeDescription 字段。它被定义为 “导致该实体出现此版本的变更”。如果我们进一步检查 changeDescription 的属性，就可以看到它是如何存储实体当前版本和最后版本之间的差异的。

这样就可以看到每个实体实例的最后更新步骤。不过，有时这种程度的跟踪可能还不够。

OpenMetadata 的最大特点之一就是能够跟踪所有实体版本。每一个导致变更的操作（PUT、POST、PATCH）都会产生一个跟踪记录，并存储在 change_event 表中。

使用 API 获取事件数据，或直接探索每个实体的不同版本，都能为数据消费者和生产者提供强大的调试能力。

## 应用程序接口组件图

现在我们已经清楚地了解了主要组件及其作用，接下来我们将分析对 API 进行 POST 和 PUT 调用的逻辑流程。本节的主要目的是熟悉代码组织及其主要步骤。

为充分利用本节内容，建议您同时关注源代码，从您想用作示例的实体 JSON 到资源和存储库的实现。

### 创建新实体--POST

我们将从最简单的情况开始： 通过 POST 调用创建一个新实体。这是审查的第一个要点，因为部分逻辑和方法会在更新过程中重复使用。

![create-new-entity](../../../images/v1.5/main-concepts/high-level-design/create-new-entity.png)

#### 创建

我们已经知道，HTTP 调用的接收方将是 EntityResource。在这里，我们有带 @POST 注解的创建函数，以及对 API 端点和预期模式的描述。

这第一个组件的作用是接收调用并验证请求正文和报头，但真正的实现是在实体资源库(EntityRepository)中进行的，我们已经将其描述为 DAO。对于 POST 操作，内部流程非常简单，由两个步骤组成：

- 准备(Prepare)： 在服务器端验证实体数据并计算一些属性。
- 存储(Store)： 将实体 JSON 及其关系保存到后端数据库。

#### 准备

该方法用于在 POST、PUT 和 PATCH 操作中验证要创建的实体，并为实体准备好所有需要的属性和关系。

例如，我们在此处理根据实体的层次结构设置实体的 FQN 的过程。虽然所有实体都需要 FQN，但这并不是我们希望在请求中收到的属性。

此外，我们还会检查接收到的属性是否正确，例如，我们是否有一个有效的`User`作为`owner`，或`Table`是否有一个有效的`Database`。

#### 存储

存储过程分为两个不同的步骤（因为我们有两个表来保存信息）。

我们将经过验证的实体从任何 href 属性（如`owner`或`tags`）中剥离，以便只存储包含实体内在值的 JSON 文档。

然后，我们为上面省略的属性存储关系图表示法。

在这些调用结束时，我们会得到一个经过验证的实体，该实体包含所有必需的属性，这些属性都经过验证并相应地存储。然后，我们可以将创建的实体返回给调用者。

### 创建或更新实体-PUT

现在，让我们以 POST 讨论中学到的知识为基础，将示例扩展为处理 PUT 请求。

最初的步骤非常相似：

- 我们的资源`Resource`中有一个注释为 @PUT 的函数，用于处理头、认证和模式。
- 然后，资源`Resource`调用存储库的 DAO，引导与数据相关的逻辑。
- 在准备步骤中，我们会验证实体并处理一些属性。

处理并验证实体请求后，我们会通过实体的 FQN 查询后端数据库，检查实体实例是否已经存储。如果没有，我们就会按照与 POST 操作相同的逻辑-->简单创建。否则，我们需要验证更新的字段。

![create-or-update](../../../images/v1.5/main-concepts/high-level-design/create-or-update.png)

#### 设置字段

我们不能允许更新给定实体实例的所有字段。例如，实例创建后，id 或名称将保持不变，表的数据库也是如此。

可以更改的指定字段列表是在每个实体的存储库中定义的，我们应该只允许更改那些在对象的整个生命周期中可以自然演变的属性。

在这一步中，我们为实体设置 JSON 模式定义（如 MlModel 的算法）所要求的字段，或者在 GET 操作中，以 `GET <url>/api/v1/<collectionName>/<id>?fields=field1,field2...` 的方式请求的字段。

#### 更新

实体库(EntityRepository)中有一个实体更新器接口(EntityUpdater)的抽象实现，负责定义所有实体通用的通用更新逻辑流。

更新调用中处理的主要步骤是

1. 更新实体的通用字段，如描述或所有者。
2. 运行实体的特定更新，这些更新由每个实体的 EntityUpdater 扩展实现。
3. 将更新后的实体 JSON 文档存储到 MySQL 的实体表中。

#### 实体特定更新

每个实体都有一组定义它的属性。这些属性会有非常具体的行为，因此更新逻辑的实现由每个实体存储库负责。

例如，我们可以更新表中的列，或者更新包含 MlModel 性能指标的仪表盘。
就实体如何在内部执行更新、实体版本如何受到影响或对关系数据的影响而言，这两种更改都会得到不同的处理。

为了便于讨论，我们将遵循几个更新方案。

#### 示例 1 - 更新表的列

更新列时，我们需要比较原始实体中的现有列集与 PUT 请求中接收到的列。

如果我们接收的是现有列，我们可能需要更新其描述或标记。这种更改将被视为次要更改。因此，实体的版本将按照软件发布规范模型提升 0.1。

但是，如果在更新的实例中没有收到存储的列，会发生什么情况呢？这意味着该列已被删除。这种类型的更改可能会破坏表数据上的集成。
因此，我们可以将这种情况标记为重大更新。在这种情况下，实体的版本将增加 1.0。

查看变更事件或访问实体历史会很容易地向我们展示实体实例的演变过程，这在调试数据问题时非常有价值。

#### 示例 2 - 更新 ML 模型的仪表板

MlModel 的属性之一是指向仪表盘的 EntityReference，该仪表盘记录了其性能指标的演变。

由于该属性是对另一个现有实体的引用，因此该数据不会直接存储在 MlModel JSON 文档中，而是以关系图的形式存储，正如我们之前所讨论的那样。因此，在更新步骤中，我们需要

1. 如果原始实体没有 Dashboard 信息，则插入关系；
2. 如果仪表盘已被删除，则删除关系，或；
3. 如果我们现在指向了不同的仪表盘，则更新关系。

请注意，在 POST 操作中，我们需要始终调用存储关系（storeRelationship）函数，因为这是我们第一次存储实例的信息。在更新过程中，如果实体的特定属性需要，我们只需修改关系数据。

## 处理事件

在所有这些讨论和示例中，我们一直在展示后端 API 如何处理 HTTP 请求，以及实体的数据生命周期是怎样的。我们不仅关注 JSON 文档和关系，还时不时地谈到了变更事件。

此外，在 API 容器图中，我们绘制了一个容器，它代表了保存更改事件数据的表，但我们还没有发现任何访问它的组件。

这是因为 API 服务器是由 Jetty 提供支持的，这意味着幸运的是我们不需要自己进行这些调用！
通过定义一个 ChangeEventHandler 并在创建服务器时将其注册，调用的后处理过程将以透明的方式进行。

我们的 ChangeEventHandler 将检查实体是否已被创建、更新或删除，并将相应的 ChangeEvent 数据从我们的响应存储到后台数据库中。
